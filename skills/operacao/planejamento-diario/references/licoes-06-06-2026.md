## 1. Timezone: UTC vs Local

### Sintoma

Os DIARIOs dos agentes registravam timestamps com 4 horas de diferença do relógio local. Quando o relógio do Rafael marcava **18:00 (America/Campo_Grande, UTC-4)**, os DIARIOs mostravam **22:00**.

```
Relógio local:   18:00 BRT (-04)
DIARIO mostra:   22:00 UTC
                  ↑ 4 horas de diferença
```

### Causa Raiz

O Hermes Agent usa `timezone` do `config.yaml` para registrar timestamps. Dos 6 perfis da equipe Mac:

| Perfil | timezone | Status |
|--------|----------|--------|
| dalinar | `""` (vazio) | ❌ |
| navani | `""` (vazio) | ❌ |
| shallan | `""` (vazio) | ❌ |
| kaladin | **(ausente)** | ❌ |
| pattern | **(ausente)** | ❌ |
| jasnah | **(ausente)** | ❌ |

**3 perfis** tinham `timezone: ''` (string vazia) e **3 perfis** nem tinham a chave `timezone` no config.yaml. O Hermes default é **UTC** quando o campo está vazio ou ausente.

```
config.yaml (ANTES):
  timezone: ''         ← vazio = UTC
  # ou chave ausente  ← ausente = UTC
```

### Correção

Setar `timezone: America/Campo_Grande` em todos os 6 perfis.

```bash
# PARA CADA UM dos 6 perfis, verificar estado atual:
grep "timezone:" ~/.hermes/profiles/*/config.yaml

# Saída esperada (ANTES):
# dalinar: timezone: ''
# navani: timezone: ''
# shallan: timezone: ''
# kaladin: (linha ausente)
# pattern: (linha ausente)
# jasnah: (linha ausente)

# CORREÇÃO — se o campo existe mas está vazio:
sed -i '' 's/timezone: .*/timezone: America\/Campo_Grande/' \
  ~/.hermes/profiles/dalinar/config.yaml \
  ~/.hermes/profiles/navani/config.yaml \
  ~/.hermes/profiles/shallan/config.yaml

# CORREÇÃO — se o campo não existe, adicionar manualmente
# Abrir config.yaml e adicionar: timezone: America/Campo_Grande
```

### Verificação

```bash
grep "timezone:" ~/.hermes/profiles/*/config.yaml

# Saída esperada (DEPOIS):
# /Users/rafaelfae/.hermes/profiles/dalinar/config.yaml:timezone: America/Campo_Grande
# /Users/rafaelfae/.hermes/profiles/navani/config.yaml:timezone: America/Campo_Grande
# /Users/rafaelfae/.hermes/profiles/shallan/config.yaml:timezone: America/Campo_Grande
# /Users/rafaelfae/.hermes/profiles/kaladin/config.yaml:timezone: America/Campo_Grande
# /Users/rafaelfae/.hermes/profiles/pattern/config.yaml:timezone: America/Campo_Grande
# /Users/rafaelfae/.hermes/profiles/jasnah/config.yaml:timezone: America/Campo_Grande
```

### ⚠️ Atenção

Após alterar o `config.yaml`, o gateway precisa ser **reiniciado** para a mudança fazer efeito (veja [Problema 4](#4-gateways-com-config-desatualizada)).

### Lição Aprendida

- **NUNCA** deixe `timezone` vazio ou ausente — o Hermes não avisa que está usando UTC
- **SEMPRE** configure `timezone` no setup inicial de cada profile (veja `01-CONFIGURACAO-INICIAL.md`)
- Use `America/Campo_Grande` para Mato Grosso do Sul (UTC-4, sem horário de verão)

## 2. DIARIO.md e ESTADO-DA-EQUIPE.md Vazios

### Sintoma

Kaladin, Pattern e Jasnah estavam com DIARIOs **vazios** — nenhum registro de sessão. O `ESTADO-DA-EQUIPE.md` compartilhado também não era atualizado. Isso levou a:

1. **Perda de contexto entre sessões** — quando Rafael chamava um agente no Slack e depois no terminal, o agente não sabia o que estava acontecendo
2. **Atropelamento de tasks** — dois agentes começavam a mesma task porque nenhum consultava o estado da equipe
3. **Respostas inconsistentes** — Dalinar dava respostas diferentes na mesma thread porque não lembrava do que já tinha dito

### Causa Raiz 1: Instruções enterradas em documentação grande

As regras sobre DIARIO.md e ESTADO-DA-EQUIPE.md estavam:

- No `AGENTS.md` (~11K chars) — enterrado no meio de regras de separação de docs e checklist pré-commit
- Em skills (29K chars totais) — difícil de encontrar e lembrar

O modelo simplesmente **ignorava** as regras porque elas não estavam priorizadas no system_prompt.

### Causa Raiz 2: System_prompt não mencionava DIARIO

O `system_prompt` dos agentes (configurado no `config.yaml`) dizia apenas:

```yaml
system_prompt: "Você é {nome}. Fale em pt-BR.\n"
```

Sem menção ao protocolo DIARIO, o modelo não "ouvia" a regra como prioridade. O system_prompt é o primeiro e mais forte sinal de comportamento para o modelo — se não está lá, não existe.

### Causa Raiz 3: Caminhos relativos vs absolutos

Quando as instruções usavam caminhos relativos como `operacional/DIARIO.md`, cada agente resolvia a partir do **project dir** (`/Users/rafaelfae/Dev/oeste-gestao/`), não do **profile dir** (`~/.hermes/profiles/{agente}/`). Resultado: ninguém encontrava o arquivo.

### Correção: Injetar PROTOCOLO DIARIO no system_prompt

Adicionar o protocolo COMPLETO no `system_prompt` de cada um dos 6 agentes, com **caminhos ABSOLUTOS**.

#### Template do que adicionar ao system_prompt

```
\n\n### PROTOCOLO DIARIO — OBRIGATÓRIO (Rafael 06/06/2026)\n
Você TEM dois arquivos operacionais que DEVE usar em TODA sessão:\n
\n
1. DIARIO.md (seu diário pessoal)\n
   Local: ~/.hermes/profiles/{AGENTE}/operacional/DIARIO.md\n
   LEIA no início de cada sessão\n
   ATUALIZE ao concluir/pausar uma task\n
\n
2. ESTADO-DA-EQUIPE.md (memória coletiva)\n
   Local: ~/.hermes/profiles/dalinar/operacional/ESTADO-DA-EQUIPE.md\n
   LEIA antes de qualquer ação\n
   ATUALIZE ao iniciar/concluir/pausar\n
\n
REGRA ABSOLUTA: Se você NÃO leu o ESTADO-DA-EQUIPE antes de agir,\n
você pode estar atropelando outro agente. Sempre leia primeiro.\n
```

⚠️ **Atenção ao formato YAML!** Cada perfil tem um formato de escape diferente (veja referência `system-prompt-yaml-escape-profiles.md`):

| Perfil | Formato do system_prompt | Estratégia de injeção |
|--------|--------------------------|-----------------------|
| dalinar | Multilinha com `\ ` (YAML backslash continuation) | Substituir a linha `\ pt-BR."` |
| navani | Linha única com `\\n` (dupla barra = YAML literal) | `replace('pt-BR.\\\\n"', 'pt-BR.\\\\n\\\\n###...\\\\n"')` |
| shallan, kaladin, pattern, jasnah | Linha única com `\n` (barra simples) | `replace('pt-BR.\\n"', 'pt-BR.\\n\\n###...\\n"')` |

#### Script Python para injeção segura

```python
#!/usr/bin/env python3
"""Injetar PROTOCOLO DIARIO no system_prompt de todos os perfis."""

PERFIS = {
    'dalinar': {'formato': 'multilinha'},
    'navani':  {'formato': 'dupla_barra'},
    'shallan': {'formato': 'barra_simples'},
    'kaladin': {'formato': 'barra_simples'},
    'pattern': {'formato': 'barra_simples'},
    'jasnah':  {'formato': 'barra_simples'},
}

PROTOCOLO = (
    '\\n\\n### PROTOCOLO DIARIO — OBRIGATÓRIO (Rafael 06/06/2026)'
    '\\nVocê TEM dois arquivos operacionais que DEVE usar em TODA sessão:'
    '\\n\\n1. DIARIO.md (seu diário pessoal)'
    '\\n   LEIA no início de cada sessão'
    '\\n   ATUALIZE ao concluir/pausar uma task'
    '\\n\\n2. ESTADO-DA-EQUIPE.md (memória coletiva)'
    '\\n   Local: ~/.hermes/profiles/dalinar/operacional/ESTADO-DA-EQUIPE.md'
    '\\n   LEIA antes de QUALQUER ação'
    '\\n   ATUALIZE ao iniciar/concluir/pausar'
    '\\n\\nREGRA ABSOLUTA: leia o ESTADO-DA-EQUIPE antes de agir.'
)

for agente, config in PERFIS.items():
    path = f'/Users/rafaelfae/.hermes/profiles/{agente}/config.yaml'
    with open(path, 'r') as f:
        content = f.read()

    if config['formato'] == 'multilinha':
        # Dalinar: substituir " pt-BR.\"" adicionando protocolo antes
        old = ' pt-BR."'
        new = ' \\n' + PROTOCOLO + '"'
        content = content.replace(old, new, 1)
    elif config['formato'] == 'dupla_barra':
        # Navani: \\n no arquivo
        old = 'pt-BR.\\\\n"'
        new = 'pt-BR.' + PROTOCOLO + '\\\\n"'
        content = content.replace(old, new, 1)
    else:
        # Demais: \n simples
        old = 'Fale em pt-BR.\\n"'
        new = 'Fale em pt-BR.' + PROTOCOLO + '\\n"'
        content = content.replace(old, new, 1)

    with open(path, 'w') as f:
        f.write(content)

print("✅ Protocolo injetado em todos os 6 perfis")
```

#### Verificação

```bash
for p in dalinar navani shallan kaladin pattern jasnah; do
  if grep -q "PROTOCOLO DIARIO" ~/.hermes/profiles/$p/config.yaml; then
    echo "$p: OK"
  else
    echo "$p: AUSENTE ❌"
  fi
done
```

### Lição Aprendida

- **O system_prompt é o único lugar confiável** para regras operacionais críticas. AGENTS.md e skills são complementares, não substitutos.
- Use **caminhos ABSOLUTOS** sempre que referenciar arquivos fora do project dir. Caminhos relativos são resolvidos de forma imprevisível.
- Cada perfil Hermes pode ter formatos de escape YAML **diferentes** para o mesmo campo. Sempre verificar com `repr()` antes de fazer substituições.
- Após modificar o system_prompt, **reiniciar o gateway** (veja Problema 4).

## 3. Popup de Permissão macOS (sandboxd)

### Sintoma

Popup do macOS aparecia repetidamente durante execução de tasks:

> "Python would like to access data from other apps"
> "Python would like to access files in your Documents folder"

Isso travava tasks automatizadas que dependiam de ferramentas como ripgrep.

### Diagnóstico com log do TCC

O macOS mantém um log detalhado do subsistema TCC (Transparency, Consent, and Control):

```bash
# Capturar eventos de permissão em tempo real (precisa de Terminal com permissão)
log stream --predicate 'subsystem == "com.apple.TCC"' --style compact

# Filtrar por processo específico
log stream --predicate 'subsystem == "com.apple.TCC" AND message CONTAINS "python3"'
```

### Causa Raiz: Cadeia Python → ripgrep → sandboxd

```
Python3.12 (Homebrew)
    ↓ chama subprocess
ripgrep (rg)  ← ferramenta de busca de arquivos
    ↓ tenta acessar
~/Desktop, ~/Documents, ~/Downloads  ← áreas protegidas pelo sandbox
    ↓
sandboxd BLOQUEIA → popup de permissão
```

A ferramenta `search_files` (que usa ripgrep internamente) precisa escanear diretórios para encontrar arquivos. Quando esses diretórios estão em áreas protegidas (Desktop, Documents, Downloads), o sandbox do macOS bloqueia e exibe o popup.

### Solução

**System Settings → Privacy & Security → Full Disk Access → adicionar Terminal.app**

```
Preferências do Sistema
  └── Privacidade e Segurança
       └── Acesso Total ao Disco
            └── [+] Terminal.app   ← ativar
```

**Alternativa:** System Settings → Privacy & Security → Files and Folders → permitir que Terminal.app acesse Desktop, Documents, Downloads.

### Outras causas identificadas

| Causa | Cadeia | Trigger | Solução |
|-------|--------|---------|---------|
| Brave → mds → Calendário | Brave Browser → Spotlight metadata → Calendário | `browser_navigate` em tasks | Ajustar permissões de Calendário para Brave |
| DuckDB → firewall | DuckDB CLI → postgres_scanner → firewall | Primeira conexão de rede do DuckDB | Permitir rede local para Terminal |

### Notas importantes

- Popups aparecem apenas na **PRIMEIRA** vez que um processo tenta acessar um recurso protegido
- Após conceder permissão, o macOS **lembra** e não popupa de novo
- Se um **cron job** disparar em contexto diferente (sem terminal interativo), o popup **NÃO aparece** — o acesso é **negado silenciosamente**. O log do TCC mostra `AUTHREQ_RESULT: authValue=0` (negado)

### Lição Aprendida

- **Full Disk Access para o Terminal** resolve 90% dos popups de permissão em tasks
- Use `log stream --predicate 'subsystem == "com.apple.TCC"'` para diagnosticar qual processo está sendo bloqueado
- Cron jobs e serviços headless não exibem popup — falham silenciosamente. Verifique logs do TCC nesses casos
- Documente este passo no setup inicial de qualquer workstation macOS para agentes Hermes

## 4. Gateways com Config Desatualizada

### Sintoma

Mudanças no `AGENTS.md`, skills, ou `system_prompt` nunca entravam em vigor. Rafael editava os arquivos de configuração, mas os agentes continuavam agindo como antes.

### Causa Raiz

O Hermes Agent carrega configurações no momento em que o **gateway** é iniciado. Se o gateway continua rodando com o PID antigo, ele ignora completamente qualquer alteração feita nos arquivos depois da inicialização.

```
                   ┌─────────────────────┐
                   │   Início do dia      │
                   │  gateway.run()       │
                   │  carrega config.yaml │ ← snapshot CONGELADO
                   │  carrega skills/     │ ← snapshot CONGELADO
                   │  carrega AGENTS.md   │ ← snapshot CONGELADO
                   └──────────┬──────────┘
                              │
                              ▼
                   ┌─────────────────────┐
                   │   Durante o dia      │
                   │  Rafael edita:      │
                   │  ├ config.yaml      │ ← IGNORADO
                   │  ├ skills/*         │ ← IGNORADO
                   │  └ AGENTS.md        │ ← IGNORADO
                   └─────────────────────┘
```

### Correção

Reiniciar o gateway com `--replace` para recarregar todas as configurações:

```bash
# Para CADA perfil que teve configuração alterada:
hermes --profile dalinar gateway run --replace
hermes --profile navani gateway run --replace
hermes --profile shallan gateway run --replace
hermes --profile kaladin gateway run --replace
hermes --profile pattern gateway run --replace
hermes --profile jasnah gateway run --replace
```

### Verificação

Comparar PID antes e depois do restart:

```bash
# ANTES — capturar PID do gateway
cat ~/.hermes/profiles/dalinar/gateway.pid
# Saída: 24107

# DEPOIS — executar o --replace
hermes --profile dalinar gateway run --replace

# Verificar novo PID
cat ~/.hermes/profiles/dalinar/gateway.pid
# Saída: 24235 ← DIFERENTE do anterior

# Confirmar que o gateway está rodando
ps aux | grep hermes | grep -v grep
```

Também é possível verificar no log de diagnóstico:

```bash
grep "gateway.start" ~/.hermes/profiles/dalinar/logs/gateway-exit-diag.log | tail -5
# {"ts": "2026-06-06T17:16:56.515", "tag": "gateway.start", "pid": 24107, "replace": true}
# {"ts": "2026-06-06T17:30:01.236", "tag": "gateway.start", "pid": 25282, "replace": false}
```

Procure eventos `gateway.start` com `"replace": true` — esses são os que carregam as novas configurações.

### ⚠️ Atenção

- `hermes gateway run` (sem `--replace`) falha se já houver um gateway rodando
- `hermes gateway run --replace` mata o processo antigo e inicia um novo
- Se o gateway antigo foi iniciado de um diretório diferente, o `--replace` pode não encontrar o PID. Nesse caso, mate manualmente: `kill $(cat gateway.pid)` ou `pkill -f "hermes.*gateway"`

### Lição Aprendida

- **Toda mudança** em config.yaml, skills ou AGENTS.md requer **restart do gateway**
- Crie um alias ou script para reiniciar todos os 6 gateways de uma vez:

```bash
alias restart-gateways='for p in dalinar navani shallan kaladin pattern jasnah; do
  echo "Reiniciando $p...";
  hermes --profile $p gateway run --replace 2>/dev/null;
done'
```

- Documente o PID do gateway como parte do DIARIO.md para facilitar debug

## 5. Pipes Duplicados em INDICE.md/PLANO.md

### Sintoma

Tabelas markdown apareciam com pipes duplicados:

```
ANTES (correto):
| 07 | task_07 | Navani | ClinicaConfig | K1 | ✅ | 👁 | f7ec31d |

DEPOIS (corrompido):
|| 07 | task_07 | Navani | ClinicaConfig | K1 | ✅ | 👁 | f7ec31d |
```

O `||` no início da linha quebra a renderização da tabela markdown — o parser interpreta como coluna extra ou linha inválida.

### Causa Raiz: `patch` tool com fuzzy matching

Quando a ferramenta `patch` é usada para substituir linhas em tabelas markdown, o algoritmo de **fuzzy matching** (que tenta achar correspondência mesmo com pequenas diferenças) às vezes adiciona um pipe extra no início da linha de substituição.

```
Contexto do patch:
  |---|---------|--------|-----------|:--:|:--:|:--:|:------:|
  | 06 | task_06 | Navani | Clinica model | K1 | ✅ | 👁 | af0f7b6 |
  | 07 | task_07 | Navani | ClinicaConfig | K1 | ✅ | 👁 | f7ec31d |  ← alvo
  | 08 | task_08 | Navani | Admin Clinica | K1 | ✅ | 👁 | ba49e81 |

Fuzzy matching interpreta o `|` como parte do contexto anterior
e produz:
  || 07 | task_07 | Navani | ClinicaConfig | ...  ← PIPE DUPLICADO
```

### Prevenção

**NUNCA usar `patch` tool para linhas individuais de tabela markdown.** Em vez disso:

1. **Usar `sed`** para substituições pontuais em tabelas:

```bash
# Correção: remover pipes duplicados
sed -i '' 's/^|| /| /g' INDICE.md

# Exemplo de substituição de task específica com sed
sed -i '' 's/| 07 | task_07 |.*/| 07 | task_07 | Navani | ClinicaConfig | K1 | ✅ | 👁 | f7ec31d |/' INDICE.md
```

2. **Usar `read_file` + `write_file`** para blocos grandes: leia a seção, edite como string Python e escreva de volta.

3. **Se precisar usar patch, inclua contexto MUITO amplo** — pelo menos 3 linhas antes e depois da linha alvo, para evitar ambiguidade.

### Verificação

Após qualquer edição em tabelas markdown, sempre conferir:

```bash
# Verificar se há pipes duplicados
grep -n '^||' INDICE.md PLANO.md

# Verificar se todas as linhas de tabela começam com pipe único
grep -n '^|' INDICE.md | head -20
# Cada linha deve começar com exatamente UM pipe
```

### Lição Aprendida

- **`patch` tool + tabelas markdown = risco de corromper pipes.** O fuzzy matching é ótimo para código, mas problemático para pipes.
- **Prefira `sed` para linhas de tabela.** A regex é previsível e não tem fuzzy matching.
- **Sempre verifique** o resultado de patches em markdown com `read_file` ou `grep` visual.
- Este problema já foi registrado nos Pitfalls do SKILL.md (`planejamento-diario`):
  > "Pipe duplicado em patches de tabelas: Ao usar `patch` em tabelas markdown, matching fuzzy pode adicionar `|||`. Preferir `patch` com blocos grandes de contexto ou `sed`/`perl`. Sempre verificar com `read_file` apos patch."
